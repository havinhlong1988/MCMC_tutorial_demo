**1. Switching between gradient to pure layercake:** <code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">CALgroup.C/updategroup</code>
<p style="text-align: justify;">
This function is used to update the model used in the inversion process,
which is internally represented in <strong>layer-cake style</strong> (parameterized
as stacked layers with thickness and velocity). It controls the
transformation from different input 1D model types (e.g., B-spline,
layer-cake, gradient) into the layer-cake representation:
</p>
<ol start="1" style="text-align: justify;">
  <li>
    <code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">Updategroup1</code>: convert from layer-cake like model to
    layer-cake–like model
  </li>
</ol>

<ol start="2" style="text-align: justify;">
  <li>
    <code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">Updategroup2</code>: convert from B-spline like model to
    layer-cake–like model
  </li>
</ol>
<ol start="3" style="text-align: justify;">
  <li>
    <code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">Updategroup3</code>: convert from gradient like model to
    layer-cake–like model
  </li>
</ol>
<ol start="4" style="text-align: justify;">
  <li>
    <code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">Updategroup4</code>: convert from water layer model to layer-cake–like
    model (*I am not sure what is this, but it may be used to handle
    inversion for models where a water layer exists on top of the
    structure.*)
  </li>
</ol>

Current setup for input layer-cake model and MCMC process, the function
<code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">Updategroup1</code> calculate the velocity as linear interpolation velocity
between two control points:

<div style="text-align: left; margin-left: 1px;">
$$
V(z)=V_i+\frac{V_{i+1}-V_i}{Z_{i+1}-Z_i}(Z-Z_i)\tag{3.1}
$$
</div>

Where:

- ***V(z)*** : velocity at depth ***z***

- ***V<sub>i </sub>***and ***V<sub>i+1</sub>***: velocity at control
  depth ***i*** and ***i+1*** with ***z*** located between them)

- ***z*** : depth between layers.

So it is become gradient style model. If you want to change back to
layer-cake style, commend like for gradient style and uncommend the like
for pure layer-cake. Then the ***V(z)*** is simply as”

<div style="text-align: left; margin-left: 1px;">
$$     
 V(z) = V_{i}  \tag{3.2}
$$
</div>


or

<div style="text-align: left; margin-left: 1px;">
$$                                            
 V(z) = \frac{\left( V_{i} + V_{i + 1} \right)}{2}   \tag{3.3}
$$
</div>

**2. How the misfit of the model is computed**: <code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">CALmodel.C/compute_misfit</code>

The misfit function measures the difference between the observed data
and the synthetic data predicted from the current model. For each data
point, the residual is first normalized by the data uncertainty:

a\. The <span style="color:purple"><strong>residual</strong></span> of data point *ith* is calculate as:

<div style="text-align: left; margin-left: 1px;">
$$                                                          
 r_{i} = \frac{\left( d_{obs,i} - d_{syn,i} \right)}{\sigma_{i}}  \tag{3.4}
$$
</div> 

where:

- ***d<sub>obs </sub>***: observed data

- ***d<sub>syn </sub>***: synthetic data from the model

- **$\sigma_i$**: uncertainty of the *i*-th data point

b\. The <span style="color:purple"><strong>misfit</strong></span> of each dataset then calculate as:

<div style="text-align: left; margin-left: 1px;">
$$
{misfit}_{data} = \sqrt{\frac{1}{N}\sum_{i = 1}^{N}\left( r_{i} \right)^{2}}  \tag{3.5}
$$
</div> 

This means the misfit is essentially the **root-mean-square normalized
residual**. The normalization by the number of data points is important
because it prevents datasets with more points from automatically giving
a larger misfit just because they contain more samples.

c\. After the individual <span style="color:purple"><strong>misfit</strong></span> are computed, they are combined into a
<span style="color:purple"><strong>total misfit</strong></span> using <strong>weighting</strong> factors:

<div style="text-align: left; margin-left: 1px;">
$$
m_{total} = w_{ph}m_{ph} + w_{gv}m_{gv} + w_{hv}m_{hv} + w_{rf}m_{rf}  \tag{3.6}
$$
</div> 
where $w_{ph}, w_{gv}, w_{hv}, w_{rf}$ denote the weights assigned to each dataset.

If equal <strong>weighting</strong> is chosen, the code distributes the weights equally
among all available data types. If manual <strong>weighting</strong> is used, the user
defines the weight for each data type, and the total should sum to
 <span style="color:blue"><strong>1</strong></span>.

d\. <span style="color:purple"><strong>Likelihood function</strong></span>: The <span style="color:purple"><strong>likelihood function</strong></span> converts the <span style="color:purple"><strong>misfit</strong></span>
into a probability-like value used in the <span style="color:darkgreen"><strong>MCMC sampling</strong></span>. In general, the
likelihood decreases as the <span style="color:purple"><strong>misfit</strong></span> increases. A common form is:

<div style="text-align: left; margin-left: 1px;">
$$
L = \exp\left( - \frac{1}{2}S \right) = \exp\left( - \frac{1}{2}\sum_{i = 1}^{N}\left( r_{i} \right)^{2} \right)  \tag{3.7}
$$
</div> 

Then the likelihook for all the dataset express as:

<div style="text-align: left; margin-left: 1px;">
$$
L = exp\left( - \frac{1}{2}\left\lbrack w_{ph}S_{ph} + w_{gv}S_{gv} + w_{hv}S_{hv} + w_{rf}S_{rf} \right\rbrack \right)  \tag{3.8}
$$
</div> 

This means:

- smaller misfit $\rightarrow$ larger likelihood

- larger misfit $\rightarrow$ smaller likelihood

So models that fit the data better are more likely to be accepted during
the MCMC process.

**3. Model quality control:** <code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">CALmodel.C/goodmodel</code>

This option is activated when the last row of
<span style="color:red; font-family:'Courier New', monospace; font-weight:bold;">{Project}/data/{sta}_data/in.connector</span> is set to <span style="color:blue;"><strong>1</strong></span>. This setting
can be controlled by the parameter <span style="color:blue;"><strong>MC_QC_thres</strong></span> in
<span style="color:red; font-family:'Courier New', monospace; font-weight:bold;">SetupData/parameter.py</span>. when it active, the function
<code style="color:red; font-family:'Courier New', monospace; font-weight:bold;">CALmodel.C/goodmodel</code> is used to constrain the data quality and model
behavior as follows:

1.  **Check valid forward data (Vph, Gv, HV, RF)**: If any forward
    result is abnormal, the model is rejected immediately.

2.  **Check monotonic increase of Vs in sediment**: So if a deeper
    sediment sublayer has lower Vs than the shallower one, the model is
    rejected.

3.  **Check monotonic increase of Vs in crust**: So if a deeper crustal
    sublayer has lower Vs than the shallower one, the model is rejected.

4.  **Check monotonic increase of Vs in mantle**: So if a deeper mantle
    sublayer has lower Vs than the shallower one, the model is rejected.

5.  **Check positive jump at sediment–crust boundary**: The code checks
    the interface between sediment and crust. It requires the bottom
    sediment velocity to be smaller than the top crust velocity. If
    bottom sediment is greater than top crust, the model is rejected.

6.  **Check positive jump at crust–mantle boundary**: The code also
    checks the interface between crust and mantle. It requires the
    bottom crust velocity to be smaller than the first mantle velocity.
    If bottom crust is greater than the mantle top, the model is
    rejected.

7.  **Check monotonic increase in sediment layers (use for MCMC in
    bspline style only)**: If sediment exists, the code checks that
    sediment Vs also increases with depth.

8.  **Check consistent ordering of spline/control points (use for MCMC
    in bspline style only)**: There is a QC check for spline-style
    models: (<span style="color:blue;"><strong>1</strong></span>) the second crust spline should not be smaller than the
    first. (<span style="color:blue;"><strong>2</strong></span>) the second mantle spline should not be smaller than the
    first. So the code is also checking that the top part of the spline
    model has a positive or non-negative trend.

9.  **Check positive velocity gradient at top of selected layers (use
    for MCMC in bspline style only)**: The code uses the vector vgrad to
    apply an additional positive-gradient check for selected groups.
    This means, for the groups listed in vgrad, the first two velocity
    values must satisfy.
